How Email Approval Works

The Email Approval system automatically ingests marketing emails from Gmail, lets you assign them to clients, and sends approval requests to designated approvers. The full flow:

  1. Ingest — The Gmail poller checks for unread emails every 60 seconds (or manually via Poll Now). Emails matching subject filters are pulled in and sanitized for fast viewing.
  2. Assign — An admin assigns the email to a client from the Email Queue. This creates approval records for each of the client's configured approvers.
  3. Notify — Click Send for Approval to email (and optionally SMS) each approver a unique, one-time approval link.
  4. Review — Approvers click the link to view the email, see other approvers' status, leave comments, and approve or reject. No login required.
  5. Resolve — Once all required approvers respond, the email status updates to Approved or Rejected automatically.

The portal uses a Google Service Account with Domain-Wide Delegation (DWD) to read and send email on behalf of any user in your Google Workspace domain.

One-time setup:
  1. In Google Cloud Console, create a Service Account and download the JSON key.
  2. Enable the Gmail API for your project.
  3. In Google Workspace Admin → Security → API Controls → Domain-wide Delegation, add the service account's Client ID with scope: https://mail.google.com/
  4. Set the service account JSON via one of:
    • GOOGLE_SERVICE_ACCOUNT_JSON — raw JSON string (for production)
    • GOOGLE_SERVICE_ACCOUNT_FILE — path to JSON file (for local dev)
  5. In Portal Settings, set GMAIL_ADDRESS to the inbox to monitor (e.g. support@yourdomain.com).
Per-client sender:

Each client can have a different From Name and From Email. The portal impersonates that email address via DWD when sending approval notifications — no Send-As aliases needed. The From Email must be a real mailbox in your Google Workspace domain.

  1. Go to Admin → Clients and create a client.
  2. Set the Subject Filter — the word to match in email subjects (e.g. malliotakis). The poller will only ingest emails matching at least one client's filter.
  3. Set From Name and From Email — the sender identity for approval notification emails to this client's approvers.
  4. Add Approvers to the client:
    • Portal user — link to an existing portal account (can also vote from the admin panel).
    • External — provide name, email, and optionally a phone number for SMS. They approve via the unique link only.
    • Mark approvers as Required or Optional. All required approvers must approve for the email to be marked Approved. Any single rejection marks it Rejected.

SMS notifications are optional. If configured, approvers with a phone number will receive an SMS alongside the email.

  1. Create a Twilio account.
  2. Get a phone number from the Twilio console.
  3. Complete A2P 10DLC registration (required for US SMS delivery): register your Brand and create a Campaign in Twilio's Messaging → Trust Hub.
  4. In Portal Settings, set:
    • TWILIO_ACCOUNT_SID
    • TWILIO_AUTH_TOKEN
    • TWILIO_PHONE_NUMBER (E.164 format, e.g. +12125551234)
  5. Use the Test SMS button on the Settings page to verify delivery.

If Twilio is not configured, SMS is silently skipped — email notifications always go out.

  1. Check the Email Queue — new emails appear with "Pending" status. Click one to open the detail view.
  2. Assign to a client — use the dropdown to assign. This auto-creates approval records for the client's approvers.
  3. Send for Approval — click the button. Each pending approver gets an email (and SMS if configured) with a unique link.
  4. Monitor — the detail page shows real-time approval status. Comments from approvers appear in the thread.
  5. Done — once all required approvers respond, the email moves to Approved or Rejected. Check the Approval Log for a history of all decisions.

Approvers receive an email with a Review & Approve button. Clicking it opens a page with:

  • Full email preview (left column) — the original marketing email rendered cleanly.
  • Approver status (right sidebar) — who has approved, rejected, or is still pending.
  • Comments thread — shared across all approvers. Anyone can post comments visible to the group.
  • Approve / Reject buttons with optional note.

No login or account is needed. Each link is unique and single-use — once a decision is submitted, the link expires.

Each client can have custom notification templates. Set them in Clients → edit a client.

Available variables:
VariableDescription
{approver_name}Recipient's name
{subject}Email subject line
{from}Original sender name or address
{client}Client name
{approve_url}Unique approval link

If no custom template is set, a clean default template is used. CSS curly braces in HTML templates are safe — only recognized variable names are replaced.
